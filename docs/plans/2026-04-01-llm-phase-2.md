# LLM Integration Phase 2 Plan

> Note: This plan is for the follow-up commit after the current configurable-provider foundation lands.

**Goal:** Harden the new LLM coach integration for production use by closing the remaining workflow, recovery, observability, and UX gaps without breaking the current hexagonal boundaries.

**Architecture:** Keep `src/domain/llm/**` provider-agnostic, keep provider wire formats inside `src/adapters/llm/**`, keep settings transport logic inside `src/adapters/rest/settings/**`, and keep workout-summary orchestration in `src/domain/workout_summary/**`. Any new recovery behavior must persist local state before external side effects and must remain idempotent across retries and restarts.

**Tech Stack:** Rust, Axum, MongoDB, reqwest, React, TypeScript, Vitest

---

## Phase 2 Scope

This phase focuses on the gaps that remain after the first end-to-end provider integration:

- recover pending coach replies after process crashes or provider timeouts
- invalidate and rotate Gemini/OpenAI/OpenRouter cache state safely when settings change
- improve provider test coverage from adapter level up to REST workflow level
- surface provider/model/cache usage more clearly in logs and persisted records
- improve the settings UX with provider defaults and better validation feedback

---

## Task 1: Make pending reply operations recoverable instead of only durable

**Goal:** Prevent `pending` coach reply operations from becoming permanent dead ends after crashes or restarts.

**Files:**
- Modify: `src/domain/workout_summary/model.rs`
- Modify: `src/domain/workout_summary/ports.rs`
- Modify: `src/domain/workout_summary/service.rs`
- Modify: `src/adapters/mongo/coach_reply_operations.rs`
- Modify: `src/main.rs`
- Add tests under: `tests/workout_summary_service/`

**Steps:**
1. Extend `CoachReplyOperation` with retry-oriented metadata such as `started_at_epoch_seconds`, `last_attempt_at_epoch_seconds`, and `attempt_count`.
2. Add repository queries for stale pending operations and for marking an operation as retryable after timeout.
3. Define a clear timeout window for “stuck pending” operations so a later request can safely resume generation.
4. Update `WorkoutSummaryService::generate_coach_reply()` so it can:
   - return completed replies immediately
   - resume stale pending work
   - avoid duplicate coach messages when persistence succeeds but the final response path fails
5. Decide whether recovery runs lazily on the next user message or through a startup repair pass in `main.rs`; prefer lazy recovery first unless startup repair is clearly needed.
6. Add focused tests for restart-like scenarios and partially completed workflows.

**Verification:**
- `cargo test --test workout_summary_service`

## Task 2: Invalidate provider cache state when configuration changes

**Goal:** Avoid reusing stale cache entries when provider, model, or stable prompt context changes.

**Files:**
- Modify: `src/domain/settings/service.rs`
- Modify: `src/adapters/mongo/llm_context_cache.rs`
- Modify: `src/adapters/llm/workout_summary_coach.rs`
- Modify: `src/adapters/rest/settings/handlers.rs`
- Add tests under: `tests/settings_rest/` and `tests/workout_summary_service/`

**Steps:**
1. Define explicit invalidation rules for cache records when:
   - `selectedProvider` changes
   - `selectedModel` changes
   - the relevant API key changes
   - coach stable context hash changes
2. Add repository operations to delete or mark obsolete cache records by user and provider/model scope.
3. Trigger invalidation from the settings update flow after the new values are persisted successfully.
4. Make the workout-summary LLM adapter prefer only valid non-expired cache records and ignore stale ones even if present.
5. Add tests proving a settings change forces a fresh provider request instead of reusing an old cache handle.

**Verification:**
- `cargo test --test settings_rest`
- `cargo test --test workout_summary_service`

**Steps:**
1. Review the normalized usage fields and add any missing provider-neutral telemetry that is already available from upstream responses.
